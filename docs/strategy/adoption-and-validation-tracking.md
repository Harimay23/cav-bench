# Adoption and Validation Tracking

## Purpose

Maintain accurate project records for external technical review, reproduction, adoption, findings, and resulting system improvements.

## Public and restricted project records

### Public repository

May include:

- public issue and PR discussions;
- contributor names and GitHub activity;
- public benchmark results;
- approved quotations;
- published case studies;
- community session links;
- public mappings and references.

### Restricted project records

May include:

- private emails;
- meeting notes;
- confidential architecture details;
- non-public benchmark outputs;
- adopter identity restrictions;
- publication permissions;
- draft statements.

Never commit confidential adopter information to the public repository.

## Validation tracking fields

| Field | Description |
|---|---|
| Date | Date of interaction or event |
| Person | Reviewer, contributor, or user |
| Organization/project | Affiliation at time of engagement |
| Reviewer role or expertise | Relevant technical or domain context |
| Engagement type | Review, run, pilot, contribution, session, citation |
| Technical scope | What was evaluated |
| Specific feedback | Exact substantive input |
| Resulting change | Code, method, scenario, architecture, or process change |
| Benchmark run | Version and configuration |
| Finding | Failure or gap identified |
| Remediation | Control or design change |
| Retest | Before-and-after result |
| Public reference | Issue, PR, article, repository, presentation |
| Publication permission | Whether identity, organization, or quotation may be published |
| Supporting artifacts | Files, screenshots, reports, correspondence |

## Validation maturity levels

### Level 1 — Engagement

An external party discusses or reviews the work.

### Level 2 — Reproduction

An external party independently runs or reproduces the work.

### Level 3 — Application

An external party applies it to a framework, workflow, or scenario.

### Level 4 — Influence

A finding causes a documented architecture, control, or testing-process change.

### Level 5 — Sustained community use

A recognized project or organization incorporates the work into recurring guidance, tooling, evaluation, or a maintained community work product.

## Case-study minimum records

- externally identifiable or credibly anonymized adopter;
- system or workflow evaluated;
- baseline testing approach;
- hidden failure detected;
- CAV evidence trace;
- recommended change;
- implementation or accepted design decision;
- retest result;
- adopter confirmation;
- publication permission for the approved level of detail.

## Claim discipline

Use only language supported by evidence.

Allowed examples:

- “Reviewed by…”
- “Reproduced independently by…”
- “Used in a controlled evaluation by…”
- “The finding led the team to…”
- “Presented to…”

Avoid unless explicitly supported:

- “Officially supported”
- “Adopted by the industry”
- “Standard”
- “Certified”
- “Endorsed by”
